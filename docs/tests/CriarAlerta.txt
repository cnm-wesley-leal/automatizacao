## M1 — Acesso e Pré-condições de Criação

---

### CT01 — Usuário logado com filtros visualiza botão "Criar Alerta"

#### Objetivo
Validar que o botão "Criar Alerta" é exibido na tela de listagem de anúncios de imóveis quando o usuário está autenticado e possui filtros aplicados.

#### Pré-Condições
- Usuário cadastrado e autenticado
- Acesso à tela de listagem de imóveis
- Ao menos um filtro aplicado na busca (finalidade + tipo + localização)

#### Passos

| Id | Ação | Resultado Esperado |
|----|------|--------------------|
| 1 | Acessar a página de listagem de imóveis com filtros aplicados (ex.: Apartamentos > Comprar > Curitiba) | A listagem é exibida com os resultados correspondentes |
| 2 | Observar o cabeçalho/heading da listagem em desktop, ou o botão flutuante acima da bottom-nav em mobile | O botão "Criar Alerta" é exibido e está habilitado |
| 3 | Verificar o estado visual do botão | Botão apresenta texto "Criar Alerta" e está clicável |

#### Resultados Esperados
- Botão "Criar Alerta" visível e habilitado na listagem de imóveis com filtros preenchidos.

#### Critérios de Aceitação
- Em desktop: botão presente no heading da listagem de busca.
- Em mobile: botão flutuante visível acima da bottom-nav.
- Botão habilitado apenas quando critérios obrigatórios (finalidade, tipo, localização) estão preenchidos.

---

### CT02 — Usuário não logado clica em "Criar Alerta" (redirecionamento para login)

#### Objetivo
Validar que usuário não autenticado é redirecionado para a tela de login ao tentar criar um alerta.

#### Pré-Condições
- Usuário não autenticado (sessão encerrada ou sem conta)
- Acesso à tela de listagem de imóveis com filtros aplicados

#### Passos

| Id | Ação | Resultado Esperado |
|----|------|--------------------|
| 1 | Acessar a listagem de imóveis sem estar logado, com filtros (finalidade + tipo + localização) | Listagem exibida normalmente |
| 2 | Clicar no botão "Criar Alerta" | Sistema redireciona o usuário para a tela de login |
| 3 | Realizar o login com credenciais válidas | Usuário é redirecionado de volta para a tela de criação de alerta com os mesmos critérios preservados (sem perda de contexto) |

#### Resultados Esperados
- Redirecionamento para login ao clicar em "Criar Alerta" sem autenticação.
- Após autenticação, o fluxo de criação é retomado com os critérios originais intactos.

#### Critérios de Aceitação
- Nenhum dado de filtro é perdido após o processo de login.
- Usuário não precisa reaplicar filtros manualmente após autenticar.

---

### CT03 — Usuário sem filtros obrigatórios preenchidos clica em "Criar Alerta"

#### Objetivo
Validar que o sistema bloqueia ou orienta o usuário quando nenhum filtro obrigatório está preenchido ao tentar criar um alerta.

#### Pré-Condições
- Usuário autenticado
- Tela de listagem de imóveis sem nenhum filtro aplicado (estado inicial)

#### Passos

| Id | Ação | Resultado Esperado |
|----|------|--------------------|
| 1 | Acessar a listagem de imóveis sem aplicar nenhum filtro | Listagem exibe todos os imóveis disponíveis |
| 2 | Tentar clicar no botão "Criar Alerta" (se visível) | Botão exibido como desabilitado, OU o sistema exibe uma tela/aviso informando que filtros obrigatórios são necessários |
| 3 | Ler a mensagem de orientação exibida | Mensagem específica indica quais critérios obrigatórios estão faltando (finalidade, tipo de imóvel, localização) |

#### Resultados Esperados
- Criação do alerta impedida quando nenhum critério obrigatório está preenchido.
- Sistema exibe orientação clara sobre quais critérios são necessários.

#### Critérios de Aceitação
- Botão "Criar Alerta" desabilitado ou a ação bloqueada sem critérios obrigatórios.
- Mensagem de orientação específica (Regra 1.7 da spec geral).

---

### CT04 — CTA de criação de alerta exibido no fim de resultados ou em página sem conteúdo

#### Objetivo
Validar que o CTA de "Criar Alerta" é apresentado ao final da listagem ou em páginas de estado vazio.

#### Pré-Condições
- Usuário autenticado
- Busca com critérios que retornam poucos ou nenhum resultado

#### Passos

| Id | Ação | Resultado Esperado |
|----|------|--------------------|
| 1 | Realizar busca de imóveis com filtros que resultem em zero ou poucos anúncios | Tela exibe estado vazio ou fim da listagem |
| 2 | Observar a tela de estado vazio ou o final da listagem | CTA "Criar Alerta" é exibido como opção de ação disponível |
| 3 | Clicar no CTA | Sistema direciona para o fluxo de criação de alerta com os filtros atuais |

#### Resultados Esperados
- CTA visível em cenários de zero resultado ou fim da listagem.
- Navegação para o fluxo de criação funciona normalmente a partir do CTA.

#### Critérios de Aceitação
- CTA aparece consistentemente em estados de zero resultados com filtros obrigatórios preenchidos.

## M2 — Validação de Critérios Obrigatórios (Imóveis)

---

### CT05 — Bloquear criação de alerta sem "Finalidade" selecionada

#### Objetivo
Validar que o sistema exige a seleção de Finalidade (Comprar ou Alugar) como critério obrigatório para criar alerta de imóvel.

#### Pré-Condições
- Usuário autenticado
- Filtros aplicados: Tipo de imóvel e Localização preenchidos, **Finalidade ausente**

#### Passos

| Id | Ação | Resultado Esperado |
|----|------|--------------------|
| 1 | Acessar a listagem de imóveis com Tipo e Localização preenchidos, mas sem selecionar Finalidade | Listagem exibida |
| 2 | Clicar em "Criar Alerta" | Sistema bloqueia a ação e exibe mensagem informando que Finalidade é obrigatória |
| 3 | Verificar a mensagem exibida | Mensagem específica identifica "Finalidade" como critério faltante |

#### Resultados Esperados
- Criação bloqueada sem Finalidade.
- Feedback claro ao usuário sobre o critério faltante.

#### Critérios de Aceitação
- Sistema identifica especificamente "Finalidade" como campo obrigatório ausente (Regra 1.7).

---

### CT06 — Bloquear criação de alerta sem "Tipo de Imóvel" selecionado

#### Objetivo
Validar que o sistema exige ao menos um Tipo de imóvel selecionado para habilitar a criação do alerta.

#### Pré-Condições
- Usuário autenticado
- Filtros aplicados: Finalidade e Localização preenchidos, **Tipo de Imóvel ausente**

#### Passos

| Id | Ação | Resultado Esperado |
|----|------|--------------------|
| 1 | Acessar a listagem com Finalidade e Localização preenchidos, sem Tipo de Imóvel | Listagem exibida |
| 2 | Clicar em "Criar Alerta" | Sistema bloqueia e informa que Tipo de Imóvel é obrigatório |
| 3 | Verificar a mensagem exibida | Mensagem identifica "Tipo de Imóvel" como critério faltante |

#### Resultados Esperados
- Criação bloqueada sem Tipo de Imóvel.

#### Critérios de Aceitação
- Sistema identifica especificamente "Tipo de Imóvel" como obrigatório (Regra 1.7).
- Pelo menos um tipo deve estar selecionado.

---

### CT07 — Bloquear criação de alerta sem "Localização" preenchida

#### Objetivo
Validar que o sistema exige ao menos um bairro ou cidade como critério de Localização para criar alerta.

#### Pré-Condições
- Usuário autenticado
- Filtros aplicados: Finalidade e Tipo de Imóvel preenchidos, **Localização ausente**

#### Passos

| Id | Ação | Resultado Esperado |
|----|------|--------------------|
| 1 | Acessar a listagem com Finalidade e Tipo preenchidos, sem Localização | Listagem exibida |
| 2 | Clicar em "Criar Alerta" | Sistema bloqueia e informa que Localização é obrigatória |
| 3 | Verificar a mensagem exibida | Mensagem identifica "Localização" como critério faltante |

#### Resultados Esperados
- Criação bloqueada sem Localização.

#### Critérios de Aceitação
- Ao menos um bairro ou cidade inteira deve estar selecionado (Regra 1.4 + Anexo Imóveis §3).

---

### CT08 — Criar alerta com critérios opcionais (sugeridos) adicionados

#### Objetivo
Validar que o sistema aceita e salva critérios sugeridos opcionais (quartos, preço, área etc.) além dos obrigatórios.

#### Pré-Condições
- Usuário autenticado
- Filtros obrigatórios preenchidos: Finalidade + Tipo de Imóvel + Localização
- Filtros opcionais adicionados: Quartos (2+), Faixa de Preço (R$200k–R$500k), Garagens (1+)

#### Passos

| Id | Ação | Resultado Esperado |
|----|------|--------------------|
| 1 | Aplicar filtros obrigatórios + opcionais na busca | Listagem refinada exibida |
| 2 | Clicar em "Criar Alerta" | Modal/tela de "Alerta de novos imóveis" é exibida |
| 3 | Verificar os critérios exibidos no modal (modo read-only) | Todos os critérios aplicados (obrigatórios + opcionais) estão listados corretamente |
| 4 | Clicar em "Salvar alerta" | Alerta criado com sucesso; tela de confirmação é apresentada |
| 5 | Verificar o nome gerado automaticamente | Nome do alerta segue padrão: "[Tipo] para [Finalidade] em [Localização]" (ex.: "Apartamentos para comprar no Batel") |

#### Resultados Esperados
- Alerta criado com todos os critérios opcionais salvos.
- Nome do alerta gerado automaticamente pelo sistema.

#### Critérios de Aceitação
- Critérios opcionais preservados no alerta sem intervenção manual do usuário.
- Nome automático correto (Regra 1.16 + Anexo Imóveis §5).

## M3 — Regras de Negócio: Limite e Duplicidade

---

### CT09 — Bloquear criação do 6º alerta (limite de 5 alertas ativos)

#### Objetivo
Validar que o sistema impede a criação de um sexto alerta quando o usuário já possui 5 alertas ativos, exibindo orientação para excluir um existente.

#### Pré-Condições
- Usuário autenticado com exatamente 5 alertas ativos criados (qualquer domínio)
- Filtros obrigatórios preenchidos na busca de imóveis

#### Passos

| Id | Ação | Resultado Esperado |
|----|------|--------------------|
| 1 | Acessar a listagem de imóveis com filtros obrigatórios preenchidos | Listagem exibida com botão "Criar Alerta" |
| 2 | Clicar em "Criar Alerta" | Sistema bloqueia a criação e exibe mensagem informando que o limite de 5 alertas foi atingido |
| 3 | Verificar a mensagem e as opções disponíveis | Mensagem orienta o usuário a excluir um alerta existente para criar um novo |
| 4 | Verificar se há atalho para a área de gerenciamento de alertas | Link ou botão direcionando para "/alertas-de-busca/" é apresentado |

#### Resultados Esperados
- 6º alerta não é criado.
- Sistema exibe mensagem de bloqueio com orientação clara (Regra 1.8).

#### Critérios de Aceitação
- Criação bloqueada ao atingir 5 alertas ativos (contagem cross-domínio: imóveis + veículos).
- Mensagem de orientação exibida com caminho para gerenciar/excluir alertas existentes.

---

### CT10 — Bloquear criação de alerta duplicado

#### Objetivo
Validar que o sistema bloqueia a criação de um alerta com os mesmos critérios exatos de um alerta já existente, oferecendo atalho para visualizá-lo.

#### Pré-Condições
- Usuário autenticado
- Já existe um alerta ativo com os critérios: Comprar + Apartamento + Batel (Curitiba)

#### Passos

| Id | Ação | Resultado Esperado |
|----|------|--------------------|
| 1 | Aplicar os mesmos filtros da busca: Comprar + Apartamento + Batel | Listagem exibida com os mesmos critérios |
| 2 | Clicar em "Criar Alerta" | Sistema detecta duplicidade e bloqueia a criação |
| 3 | Verificar a mensagem exibida | Sistema informa que já existe um alerta com esses critérios |
| 4 | Verificar o atalho disponível | Opção/link para visualizar o alerta existente é apresentada |

#### Resultados Esperados
- Alerta duplicado não é criado.
- Atalho para o alerta existente é exibido (Regra 1.9).

#### Critérios de Aceitação
- Duplicidade identificada por correspondência exata de todos os critérios preenchidos.
- Atalho funcional direcionando para o alerta existente.

## M4 — Fluxo de Criação e Confirmação

---

### CT11 — Criar alerta com sucesso (fluxo completo — happy path)

#### Objetivo
Validar o fluxo completo e bem-sucedido de criação de alerta de imóvel por usuário autenticado com critérios obrigatórios preenchidos.

#### Pré-Condições
- Usuário autenticado com menos de 5 alertas ativos
- Filtros aplicados: Comprar + Apartamento + Curitiba/Batel
- Nenhum alerta com os mesmos critérios já existe

#### Passos

| Id | Ação | Resultado Esperado |
|----|------|--------------------|
| 1 | Acessar a listagem de imóveis com os filtros aplicados | Botão "Criar Alerta" exibido e habilitado |
| 2 | Clicar em "Criar Alerta" | Modal/tela "Alerta de novos imóveis" é aberta |
| 3 | Verificar os critérios exibidos no modal (somente leitura) | Critérios aplicados na busca estão refletidos corretamente no modal |
| 4 | Verificar o nome do alerta gerado automaticamente | Nome exibido segue padrão (ex.: "Apartamentos para comprar em Curitiba") |
| 5 | Clicar em "Salvar alerta" | Alerta criado com sucesso |
| 6 | Verificar a tela de confirmação (sucesso) | Tela de sucesso é apresentada com confirmação da criação |
| 7 | Verificar as opções na tela de sucesso | Botões "Gerenciar Alertas" e "Fechar" são exibidos |

#### Resultados Esperados
- Alerta criado e salvo com todos os critérios da busca.
- Tela de sucesso exibida após confirmação (Regra 1.15).

#### Critérios de Aceitação
- Alerta aparece na lista de alertas do usuário após a criação.
- Nome gerado automaticamente sem intervenção do usuário (Regra 1.16).
- Canais (push e e-mail) configurados automaticamente conforme permissões.

---

### CT12 — Fechar modal de criação sem salvar (botão "Fechar" / ícone X)

#### Objetivo
Validar que ao fechar o modal de criação de alerta, o usuário é retornado para a listagem sem que nenhum alerta seja criado.

#### Pré-Condições
- Usuário autenticado
- Modal "Alerta de novos imóveis" aberto após clicar em "Criar Alerta"

#### Passos

| Id | Ação | Resultado Esperado |
|----|------|--------------------|
| 1 | Com o modal de criação aberto, clicar no botão "Fechar" ou no ícone "X" | Modal é fechado |
| 2 | Verificar a tela exibida após o fechamento | Usuário retorna para a listagem de anúncios com os filtros preservados |
| 3 | Verificar se algum alerta foi criado | Nenhum alerta novo consta na área de gerenciamento |

#### Resultados Esperados
- Nenhum alerta criado ao fechar o modal.
- Usuário retorna para a listagem de anúncios.

#### Critérios de Aceitação
- Estado da listagem preservado (filtros aplicados permanecem).
- Nenhum registro de alerta criado no sistema.

---

## M5 — Configuração de Canais (Push & E-mail)

---

### CT14 — Criar alerta com permissão de push já concedida

#### Objetivo
Validar que o alerta é criado com canais push e e-mail ativos automaticamente quando a permissão de push já foi concedida pelo usuário no SO.

#### Pré-Condições
- Usuário autenticado com permissão de push concedida no dispositivo/navegador
- Filtros obrigatórios preenchidos

#### Passos

| Id | Ação | Resultado Esperado |
|----|------|--------------------|
| 1 | Clicar em "Criar Alerta" e prosseguir com "Salvar alerta" | Alerta criado com sucesso |
| 2 | Acessar os detalhes do alerta em "/alertas-de-busca/" | Push: ativo; E-mail: ativo |
| 3 | Verificar que não foi exibida oferta de ativação de push | Nenhuma tela/modal de solicitação de permissão aparece (permissão já concedida) |

#### Resultados Esperados
- Push e e-mail ativos por padrão quando permissão já foi concedida (Regra 1.10).

#### Critérios de Aceitação
- Canais configurados automaticamente sem necessidade de ação adicional do usuário.

---

### CT15 — Oferta de ativação de push após criação (permissão ainda não concedida)

#### Objetivo
Validar que o sistema oferece a ativação do canal push imediatamente após a criação do alerta quando a permissão não foi concedida, sem bloquear o alerta.

#### Pré-Condições
- Usuário autenticado sem permissão de push concedida no dispositivo
- Filtros obrigatórios preenchidos

#### Passos

| Id | Ação | Resultado Esperado |
|----|------|--------------------|
| 1 | Criar o alerta clicando em "Salvar alerta" | Alerta criado com sucesso |
| 2 | Verificar o que é exibido após a confirmação | Sistema exibe oferta para ativar notificações push (após tela de sucesso) |
| 3 | Aceitar a solicitação de permissão de push | Permissão concedida; push ativado para o alerta |
| 4 | Verificar os detalhes do alerta | Push: ativo; E-mail: ativo |

#### Resultados Esperados
- Alerta criado mesmo sem permissão de push.
- Oferta de ativação de push apresentada após confirmação (Regra 1.11).

#### Critérios de Aceitação
- Recusa da permissão push não cancela o alerta; alerta persiste com e-mail ativo (Regra 1.14).
- Oferta de push apresentada a cada nova criação enquanto permissão não concedida (Regra 1.12).

---

### CT16 — Recusar ativação de push mantém alerta ativo por e-mail

#### Objetivo
Validar que ao recusar a permissão de push após a criação do alerta, o alerta permanece ativo com canal de e-mail funcional.

#### Pré-Condições
- Usuário autenticado sem permissão de push concedida
- Alerta recém-criado com oferta de push exibida

#### Passos

| Id | Ação | Resultado Esperado |
|----|------|--------------------|
| 1 | Na oferta de ativação de push, clicar em "Não" ou "Recusar" | Oferta é dispensada sem erro |
| 2 | Verificar os detalhes do alerta em "/alertas-de-busca/" | Push: desligado; E-mail: ativo |
| 3 | Confirmar que o alerta está salvo e funcional | Alerta listado normalmente com e-mail como canal ativo |

#### Resultados Esperados
- Alerta ativo exclusivamente por e-mail após recusa de push (Regra 1.14).

#### Critérios de Aceitação
- E-mail sempre ativo por padrão (Regra 1.10).
- Push desligado por padrão quando permissão não concedida.

---

### CT17 — Orientação ao usuário com push bloqueado no nível do SO

#### Objetivo
Validar que quando a permissão de push foi negada no SO (bloqueio permanente), o sistema orienta o usuário a habilitá-la nas configurações do dispositivo.

#### Pré-Condições
- Usuário autenticado com permissão de push **bloqueada** nas configurações do SO/navegador
- Alerta criado ou tentativa de ativar push na área de gerenciamento

#### Passos

| Id | Ação | Resultado Esperado |
|----|------|--------------------|
| 1 | Ao exibir oferta de push (ou ao tentar ativar push na área de gerenciamento), verificar a mensagem exibida | Sistema não apresenta o prompt nativo do navegador (pois já foi bloqueado) |
| 2 | Verificar a orientação exibida ao usuário | Mensagem instrui o usuário a habilitar notificações nas configurações do dispositivo/navegador |

#### Resultados Esperados
- Usuário orientado corretamente quando push está bloqueado no SO (Regra 1.13).

#### Critérios de Aceitação
- Sem tentativa de prompt nativo ao usuário que bloqueou a permissão.
- Mensagem de orientação clara sobre como habilitar nas configurações.

### CT11C — Double-Click Prevention em "Salvar Alerta"

#### Objetivo
Validar que o sistema previne criação de alertas múltiplos via double-click no botão "Salvar alerta", evitando duplicatas acidentais ou requisições simultâneas.

#### Pré-Condições
- Usuário autenticado com menos de 5 alertas ativos
- Modal de criação de alerta aberto com critérios obrigatórios preenchidos (Comprar + Apartamento + Curitiba)
- Nenhum alerta com os mesmos critérios já existe

#### Passos

| Id | Ação | Resultado Esperado |
|----|------|--------------------|
| 1 | Na tela de criação de alerta, posicionar cursor sobre botão "Salvar alerta" | Botão visível, habilitado e clicável |
| 2 | Executar **double-click rápido** (< 500ms entre clicks) no botão "Salvar alerta" | Apenas **1 requisição** HTTP enviada ao backend (não 2) |
| 3 | Aguardar resposta de sucesso do servidor | Exibida tela de confirmação de criação |
| 4 | Acessar `/alertas-de-busca/` e listar alertas | **Apenas 1 alerta** criado com os critérios (não duplicado) |
| 5 | Abrir DevTools > Network tab | Verificar histórico de requisições: apenas **1 POST request** detectado durante double-click |
| 6 | (Teste alternativo) Simular conexão lenta (DevTools > Throttle: Slow 3G) e repetir passos 2–5 | Mesmo resultado: apenas 1 POST, 1 alerta criado |

#### Resultados Esperados
- Apenas 1 alerta criado mesmo com double-click ou múltiplos cliques rápidos
- Botão desabilitado ou feedback visual indicando requisição em progresso

#### Critérios de Aceitação
- ✅ Apenas 1 POST request enviado durante double-click
- ✅ Botão "Salvar alerta" desabilitado/bloqueado após 1º click até resposta do servidor
- ✅ Spinner ou mensagem "Salvando..." exibida durante requisição
- ✅ Nenhum alerta duplicado criado no banco
- ✅ Comportamento funciona em conexões lentas (3G throttling)
- ✅ Backend também valida duplicidade como defesa em profundidade

#### Notas Técnicas
- Frontend: Implementar via `disabled` attribute, debounce, ou throttle
- Backend: Validação de duplicidade + transaction lock (para evitar race conditions)
- Recomendado: Usar idempotency key no request (Regra da spec de API)

#### Severidade
🟡 **MÉDIA** — Pode gerar duplicatas ou múltiplas requisições

#### Relacionado a
- Regra 1.15 (Tela de confirmação)
- Regra 1.9 (Bloqueio de duplicidade)
- Best Practice: Idempotency em APIs

| 1 | Criar **1º alerta** (Comprar + Apartamento + Curitiba) e confirmar em "Salvar alerta" | Tela de sucesso exibida |
| 2 | Verificar se oferta de push foi exibida (Regra 1.11) | Oferta de ativação de push apresentada (1ª oferta) |
| 3 | Clicar em "Não" ou "Recusar" na oferta | Oferta fechada; alerta criado com e-mail ativo, push inativo |
| 4 | Retornar para a listagem de imóveis | Usuário volta à página de busca com filtros preservados |
| 5 | Criar **2º alerta diferente** (Alugar + Casa + Salvador) e confirmar | Tela de sucesso exibida |
| 6 | Verificar se oferta de push aparece **novamente** | Oferta reaparece (2ª oferta, mesmo após rejeição anterior) |
| 7 | Clicar em "Não" novamente | Oferta fechada; 2º alerta criado |
| 8 | Criar **3º alerta** (Comprar + Imóvel Comercial + São Paulo) | Tela de sucesso exibida |
| 9 | Verificar se oferta de push reaparece | Oferta apresentada (3ª oferta — persistência validada) |
| 10 | Aceitar a oferta na 3ª criação | Permissão push solicitada ao navegador; se concedida, push ativado |
| 11 | Criar **4º alerta** sem responder à oferta | Tela de sucesso exibida |
| 12 | Verificar se oferta **desaparece** | Sem oferta (permissão já concedida na etapa 10) |

#### Resultados Esperados
- Oferta de push reaparece a cada criação até ser aceita ou opt-out global acionado
- Histórico de rejeições não persiste entre criações (oferta não é "cancelada" por spam fatigue)
- Oferta desaparece apenas quando: (a) permissão concedida, (b) opt-out global acionado, ou (c) limite de rejeições (se especificado) atingido

#### Critérios de Aceitação
- ✅ Oferta aparece consistentemente em cada criação (enquanto permissão não concedida)
- ✅ Estado de "rejeção única" **não** bloqueia futuras ofertas
- ✅ localStorage/sessionStorage reflete estado corretamente (rastreamento de permissão)
- ✅ Transição suave entre criações (sem lag ou comportamento inesperado)
- ✅ Oferta apenas desaparece quando permanentemente concedida ou opt-out global acionado
- ✅ Testes com múltiplas criações (3+) em mesma sessão

#### Notas Técnicas
- Frontend: Rastrear estado de permissão em localStorage (chave: `notifications-permission-status`)
- Backend: Log de ofertas/rejeições para análise (analytics)
- Recomendado: Implementar "soft limit" de ofertas (ex: máx 5 por sessão) para evitar spam

#### Severidade
🟡 **MÉDIA** — Regra de negócio não respeitada em cenário comum

#### Relacionado a
- Regra 1.12 (Oferta repetida a cada criação)
- Regra 1.11 (Oferta inicial de push)
- Regra 1.10 (Configuração automática de canais)

## API REST — Endpoints & Contratos

### 1. Endpoint: POST /search-alerts

**Descrição:** Cria um alerta de busca com filtros personalizados.  
**Autenticação:** Bearer Token (JWT)  
**Resposta:** 201 Created

#### Request Schema — `CreateRealtySearchAlertRequest`

```json
{
  "segment": "REALTY",
  "transactionType": 0,           // 0 = Comprar, 1 = Alugar
  "propertyTypeIds": [1, 2],      // [Apartamento, Casa, ...]
  "cityIds": [3560],               // [Curitiba, ...]
  "neighborhoodIds": [123456],     // Opcional
  "stateIds": [15],                // Opcional
  "regionIds": [999],              // Opcional (região bairro)
  "streetIds": [555],              // Opcional
  "purpose": 0,                    // 0 = Residencial, 1 = Comercial
  "minBedrooms": 2,                // Opcional
  "minParkingSpaces": 1,           // Opcional
  "minBathrooms": 1,               // Opcional
  "minPrice": 250000,              // Opcional
  "maxPrice": 500000,              // Opcional
  "minArea": 50,                   // Opcional (m²)
  "maxArea": 150,                  // Opcional (m²)
  "featureIds": [10, 20],          // Opcional (características/áreas)
  "considerTotalRent": 1,          // 0|1, se 1 usar valor aluguel total
  "url": "https://..."             // Opcional (referência de contexto)
}
```

**Validações:**
- `segment` = "REALTY" (string literal)
- `transactionType`: 0 ≤ x ≤ 1
- `propertyTypeIds`: array não vazio, min 1 elemento
- `cityIds`: array não vazio, min 1 elemento
- `purpose`: 0 ≤ x ≤ 1
- IDs numéricos: exclusiveMinimum > 0
- Preços, área: exclusiveMinimum > 0
- considerTotalRent: 0 | 1

#### Response Schema — `GetSearchAlertItemResponse`

```json
{
  "id": 42,
  "sphinxId": "user:12345",
  "criteria": {
    "transactionType": 0,
    "propertyTypeIds": [1, 2],
    "cityIds": [3560],
    "purpose": 0,
    "minBedrooms": 2,
    "transactionTypeName": "Comprar",
    "purposeName": "Residencial",
    "propertyTypeNames": ["Apartamento", "Casa"],
    "cityNames": ["Curitiba"]
  },
  "esFilters": {
    "bool": {
      "must": [...]          // Query ES gerada (Percolator format)
    }
  },
  "name": "Apartamentos 2+ quartos para comprar em Curitiba",
  "url": "https://...",
  "criteriaCount": 3,        // Contagem de critérios (padrão: 3 obrigatórios)
  "deleted": false,
  "segment": "REALTY",
  "createdAt": "2026-05-28T10:30:00Z",
  "updatedAt": null
}
```

**Campos Enriquecidos (na resposta):**
| Campo | Origem | Descrição |
|-------|--------|-----------|
| `transactionTypeName` | Mapa interno | "Comprar" ou "Alugar" |
| `purposeName` | Mapa interno | "Residencial" ou "Comercial" |
| `propertyTypeNames` | RealtyTypeService | ["Apartamento", "Casa", ...] |
| `cityNames` | CityService | ["Curitiba", "São Paulo", ...] |
| `neighborhoodNames` | NeighborhoodService | [...] |
| `stateNames` | StateService | ["Paraná", ...] |
| `featureNames` | GroupArea (repository) | ["Piscina", "Garagem", ...] |

**HTTP Codes:**
- `201 Created` — Alerta criado com sucesso
- `400 Bad Request` — Schema inválido, limite atingido, ou filtros duplicados
- `401 Unauthorized` — Token ausente ou inválido
- `403 Forbidden` — Usuário PJ (apenas PF pode criar alertas)
- `422 Unprocessable Entity` — Validação de negócio falhou (ex: dedup)
- `500 Internal Server Error` — Falha ao indexar no ES

### 3. Endpoint: GET /search-alerts/:id

**Descrição:** Busca detalhes de um alerta específico.  
**Resposta:** 200 OK

#### Response
Retorna `GetSearchAlertItemResponse` (mesmo schema que POST)

**HTTP Codes:**
- `200 OK` — Alerta encontrado
- `404 Not Found` — Alerta não existe ou foi deletado

### 4. Endpoint: DELETE /search-alerts/:id

**Descrição:** Deleta um alerta (soft delete transacional).  
**Resposta:** 200 OK (sem body)

**Operações:**
1. Marca alerta como `deleted = true` em `tb_alerta_busca`
2. Remove todos os matches em `tb_alerta_busca_match`
3. Remove documento do Percolator em Elasticsearch
4. Rollback se qualquer operação falhar

**HTTP Codes:**
- `200 OK` — Deletado com sucesso
- `404 Not Found` — Alerta não encontrado

## Banco de Dados — Schema e Comportamentos

### Tabelas Principais

#### 1. tb_alerta_busca

```sql
CREATE TABLE tb_alerta_busca (
  id INT PRIMARY KEY AUTO_INCREMENT,
  sphinx_id VARCHAR(255) NOT NULL,
  criterios JSON,                    -- Filtros originais do front
  filtros_es JSON,                   -- Query ES gerada (Percolator)
  nome VARCHAR(500) NOT NULL,        -- Nome legível gerado
  qtd_criterios INT DEFAULT 3,       -- Contagem de critérios
  deleted BOOLEAN DEFAULT FALSE,     -- Soft delete flag
  data_cadastro DATETIME DEFAULT CURRENT_TIMESTAMP,
  data_atualizacao DATETIME NULL,
  
  UNIQUE INDEX idx_sphinx_id_nome (sphinx_id, nome),  -- Dedup
  INDEX idx_sphinx_deleted (sphinx_id, deleted),
  INDEX idx_created_at (data_cadastro)
);
```

**Regras:**
- `deleted = false` por padrão
- Queries listagem filtra `WHERE deleted = 0`
- Soft delete preserva histórico para compliance LGPD
- Dedup: `(sphinx_id, nome)` único para evitar duplicatas de mesmo usuário

#### 2. tb_alerta_busca_match

```sql
CREATE TABLE tb_alerta_busca_match (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  id_alerta INT NOT NULL,
  id_cron_log INT NULL,              -- NULL = pendente, preenchido = despachado
  matched_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  data_cadastro DATETIME DEFAULT CURRENT_TIMESTAMP,
  
  FOREIGN KEY (id_alerta) REFERENCES tb_alerta_busca(id),
  INDEX idx_id_alerta (id_alerta),
  INDEX idx_id_cron_log (id_cron_log),
  INDEX idx_pendentes (id_cron_log, id_alerta)  -- Busca eficiente de pendentes
);
```

**Regras:**
- `id_cron_log = NULL` indica match pendente de despacho
- Após despacho, `id_cron_log` é preenchido (marca histórico do cron)
- Índice `idx_pendentes` otimiza query: `WHERE id_cron_log IS NULL AND id_alerta = ?`

#### 3. tb_alerta_busca_match_item

```sql
CREATE TABLE tb_alerta_busca_match_item (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  id_match BIGINT NOT NULL,
  id_anuncio INT NOT NULL,
  segment ENUM('realty', 'vehicle') NOT NULL,
  
  FOREIGN KEY (id_match) REFERENCES tb_alerta_busca_match(id) ON DELETE CASCADE,
  UNIQUE INDEX idx_match_anuncio (id_match, id_anuncio, segment)  -- Evita duplicatas
);
```

**Regras:**
- Combinação `(id_match, id_anuncio, segment)` é única
- ElasticLoader usa `INSERT IGNORE` para silenciar duplicatas
- Cascade delete quando match é removido

### Transações Críticas

#### Fluxo de Criação (API)

```
BEGIN TRANSACTION
  1. INSERT INTO tb_alerta_busca (sphinx_id, criterios, filtros_es, nome, ...)
  2. IF last_insert_id() > 0 THEN
       Indexar documento no Percolator ES: PUT alertas_imoveis/_doc/{id}
  3. IF ES fails THEN
       ROLLBACK (DELETE registro criado no passo 1)
       THROW exception
  4. ELSE
       COMMIT
```

**Pontos críticos:**
- Rollback automático se indexação ES falhar
- Sem alerta órfão (sem documento ES)
- Sem documento ES órfão (sem registro MySQL)

#### Fluxo de Exclusão (API)

```
BEGIN TRANSACTION
  1. UPDATE tb_alerta_busca SET deleted = true WHERE id = ?
  2. DELETE FROM tb_alerta_busca_match WHERE id_alerta = ?
  3. COMMIT
  4. DELETE alertas_imoveis/_doc/{id}  (fora da transação)
```

**Notas:**
- Soft delete em primeiro lugar (garante não ser listado)
- Hard delete de matches na mesma transação
- ES delete é async (falha aqui não afeta MySQL)

## Elasticsearch — Percolator Pattern

### Índices

#### Índice: `alertas_imoveis` (Percolator)

Armazena **queries dos usuários** como documentos (padrão invertido).

```json
{
  "settings": {
    "number_of_shards": 2,
    "number_of_replicas": 1,
    "analysis": {
      "analyzer": {
        "default": { "type": "standard" }
      }
    }
  },
  "mappings": {
    "properties": {
      "query": {
        "type": "percolator"
      },
      "alert_id": { "type": "integer" },
      "sphinx_id": { "type": "keyword" },
      "segment": { "type": "keyword" }
    }
  }
}
```

**Documento armazenado:**

```json
{
  "_id": "42",
  "query": {
    "bool": {
      "must": [
        { "term": { "finalidade1.keyword": "V" } },
        { "terms": { "id_imovel_tipo": [1, 2] } },
        { "terms": { "id_cidade_imovel": [3560] } },
        { "range": { "qtd_quartos": { "gte": 2 } } },
        { "range": { "valor_imovel": { "gte": 250000, "lte": 500000 } } }
      ]
    }
  },
  "alert_id": 42,
  "sphinx_id": "user:12345",
  "segment": "REALTY"
}
```

#### Índice: `imovel_novo` (Documentos)

Armazena **imóveis** para busca normal e percolate.

```json
{
  "mappings": {
    "properties": {
      "id": { "type": "integer" },
      "finalidade1": { "type": "keyword" },
      "id_imovel_tipo": { "type": "keyword" },
      "id_cidade_imovel": { "type": "keyword" },
      "qtd_quartos": { "type": "integer" },
      "qtd_quartos_maximo": { "type": "integer" },
      "valor_imovel": { "type": "long" },
      "valor_aluguel_total": { "type": "long" },
      "area": { "type": "double" },
      "area_maxima": { "type": "double" },
      "area_grupo": {
        "type": "nested",
        "properties": {
          "id": { "type": "integer" },
          "name": { "type": "text" }
        }
      },
      "geo_point": { "type": "geo_point" }
    }
  }
}
```

### Query Builder — Filtros → Elasticsearch

O `SearchAlertEsQueryBuilder` transforma filtros front-end em queries `bool.must`:

| Filtro Front | Campo ES | Tipo Query | Operador ES |
|-------------|----------|-----------|-----------|
| `transactionType` | `finalidade1.keyword` | `term` | = |
| `propertyTypeIds` | `id_imovel_tipo` | `terms` | IN |
| `cityIds` | `id_cidade_imovel` | `terms` + `bool.should` | OR |
| `neighborhoodIds` | `id_bairro_imovel` | `terms` | IN |
| `minBedrooms` | `qtd_quartos` / `qtd_quartos_maximo` | `range` | ≥ |
| `maxBedrooms` | `qtd_quartos_maximo` | `range` | ≤ |
| `minPrice` | `valor_imovel` | `range` | ≥ |
| `maxPrice` | `valor_imovel` | `range` | ≤ |
| `minArea` | `area` | `range` | ≥ |
| `maxArea` | `area_maxima` | `range` | ≤ |
| `featureIds` | `area_grupo` | `nested` | AND |
| `purpose` | `finalidade2.keyword` | `term` | = |

**Exemplo: Filtros do usuário**

```json
{
  "transactionType": 0,
  "propertyTypeIds": [1, 2],
  "cityIds": [3560, 3512],
  "minBedrooms": 2,
  "minPrice": 250000,
  "maxPrice": 500000
}
```

**Query ES Gerada:**

```json
{
  "bool": {
    "must": [
      { "term": { "finalidade1.keyword": "V" } },
      { "terms": { "id_imovel_tipo": [1, 2] } },
      {
        "bool": {
          "should": [
            { "term": { "id_cidade_imovel": 3560 } },
            { "term": { "id_cidade_imovel": 3512 } }
          ],
          "minimum_should_match": 1
        }
      },
      { "range": { "qtd_quartos": { "gte": 2 } } },
      { "range": { "valor_imovel": { "gte": 250000, "lte": 500000 } } }
    ]
  }
}
```

### Percolate Query — Matching

ElasticLoader executa a **percolate query** para encontrar alertas que casam com um imóvel:

```json
{
  "query": {
    "percolate": {
      "field": "query",
      "document": {
        "finalidade1": "V",
        "id_imovel_tipo": 1,
        "id_cidade_imovel": 3560,
        "qtd_quartos": 3,
        "valor_imovel": 350000,
        "area": 75.5
      }
    }
  },
  "_source": false,
  "size": 10000
}
```

**Retorno:**
```json
{
  "hits": {
    "total": 3,
    "hits": [
      { "_id": "42" },   // alert_id: 42 casou
      { "_id": "55" },   // alert_id: 55 casou
      { "_id": "89" }    // alert_id: 89 casou
    ]
  }
}
```

ElasticLoader grava 3 novos registros em `tb_alerta_busca_match` com `id_cron_log = NULL`.

## Fluxo Ponta a Ponta — Sequenciação Completa

```
┌─────────────────────────────────────────────────────────────────────────┐
│ FASE 1 — CADASTRO DO ALERTA (Frontend → API)                           │
└─────────────────────────────────────────────────────────────────────────┘

1. Usuário logado aplica filtros e clica "Criar Alerta"
   POST /search-alerts {
     segment: "REALTY",
     transactionType: 0,
     propertyTypeIds: [1],
     cityIds: [3560],
     minBedrooms: 2,
     minPrice: 250000
   }

2. API (NestJS):
   ✓ Valida JWT (usuário autenticado, tipo PF)
   ✓ Valida schema (CreateRealtySearchAlertRequest)
   ✓ Conta alertas existentes (< 5)
   ✓ Normaliza arrays (sort para dedup)
   ✓ Busca alerta duplicado: SELECT ... WHERE sphinx_id=? AND nome LIKE '%Apt%'
   ✓ Se existe: RETURN 422 (conflict)
   ✓ Gera nome legível: "Apartamentos 2+ quartos para comprar em Curitiba"
   ✓ BuildSearchAlertEsQuery → transforma em bool.must
   ✓ BEGIN TRANSACTION
   ✓ INSERT tb_alerta_busca → retorna id=42
   ✓ ES: PUT alertas_imoveis/_doc/42 { query: {...}, alert_id: 42 }
   ✓ COMMIT

3. API retorna:
   201 Created {
     id: 42,
     sphinxId: "user:12345",
     criteria: { ... },
     name: "Apartamentos 2+ quartos...",
     segment: "REALTY",
     createdAt: "2026-05-28T10:30:00Z"
   }

4. Frontend mostra confirmação e oferece push


┌─────────────────────────────────────────────────────────────────────────┐
│ FASE 2 — MATCHING (ElasticLoader batch periódico ~a cada 10 min)      │
└─────────────────────────────────────────────────────────────────────────┘

1. ElasticLoader roda via CLI:
   $ node elasticloader.js CarregaImovel

2. Carrega imóveis alterados:
   SELECT id FROM tb_imovel WHERE updated > lastRunTime

3. Para cada batch de 100 imóveis:
   ✓ Enriquece: fotos, geo_point, áreas computadas
   ✓ Bulk index em imovel_novo

4. Executa percolate para cada imóvel:
   POST imovel_novo/_search {
     query: { percolate: { field: "query", document: {...} } }
   }

5. Recebe lista de alert_ids que casaram (ex: [42, 55, 89])

6. Para cada alert_id:
   INSERT INTO tb_alerta_busca_match (id_alerta, id_cron_log, matched_at)
   VALUES (42, NULL, NOW())
   → id_cron_log = NULL indica "pendente de despacho"

7. ElasticLoader enfileira notificação (SQS)


┌─────────────────────────────────────────────────────────────────────────┐
│ FASE 3 — DESPACHO DE NOTIFICAÇÕES (Cron horário + SQS + Lambda)       │
└─────────────────────────────────────────────────────────────────────────┘

1. Cron job (a cada 1 hora):
   SELECT id FROM tb_alerta_busca
   WHERE deleted = 0 AND id IN (
     SELECT id_alerta FROM tb_alerta_busca_match
     WHERE id_cron_log IS NULL
   )

2. Para cada alerta:
   SELECT id, id_anuncio FROM tb_alerta_busca_match_item
   WHERE id_match IN (SELECT id FROM tb_alerta_busca_match WHERE ...)

3. Cron → cnm-notificator API:
   POST /notifications {
     userId: 12345,
     title: "Novo imóvel compatível!",
     body: "Apartamento 3 quartos em Curitiba",
     data: { alertId: 42, realtyId: 789 }
   }

4. cnm-notificator:
   ✓ Valida x-api-key
   ✓ Enfileira em SQS (visibilidade 300s)
   ✓ Retorna 202 Accepted

5. Lambda consumer (acionada por SQS):
   ✓ Carrega secrets (API key, App ID Pinpoint)
   ✓ Busca device tokens (DynamoDB): userId=12345 → [token1, token2]
   ✓ Para cada token:
     - Monta payload FCM (Android)
     - Monta payload APNS (iOS)
     - Envia via AWS Pinpoint SendMessagesCommand
   ✓ Registra log (DynamoDB)
   ✓ Se PERMANENT_FAILURE: desativa token

6. Aws Pinpoint roteia para FCM/APNS

7. Dispositivo recebe push notification


┌─────────────────────────────────────────────────────────────────────────┐
│ FASE 4 — GERENCIAMENTO DE ALERTAS (Usuário na área logada)            │
└─────────────────────────────────────────────────────────────────────────┘

1. GET /search-alerts
   ✓ Retorna lista paginada de alertas do usuário (deleted = 0)
   ✓ Enriquece critérios com nomes legíveis

2. GET /search-alerts/:id
   ✓ Retorna detalhe de um alerta específico

3. DELETE /search-alerts/:id
   ✓ BEGIN TRANSACTION
   ✓ UPDATE tb_alerta_busca SET deleted = true
   ✓ DELETE FROM tb_alerta_busca_match
   ✓ COMMIT
   ✓ DELETE alertas_imoveis/_doc/{id}
```
